ОПИСАНИЕ ПРОЕКТА
----------------
Симулятор умного дома на Java (Maven). Включает:
- Жильцов (Father, Mother, Son, Daughter-role, Grandfather, Cat)
- Устройства умного дома (~21 тип)
- Симуляцию с шагом 15 минут
- Паттерны: State, Chain of Responsibility, Observer, Singleton, Factory
- Визуализацию на Swing (пиксель-арт)
- Генерацию отчётов (активность, потребление, события, конфигурация)

БАГ #1 [КРИТИЧЕСКИЙ] — WeatherService.tick() никогда не вызывается
────────────────────────────────────────────────────────────────────────────────
Файл:    src/.../simulation/SimulationEngine.java
Метод:   runStep()

ОПИСАНИЕ:
  В методе runStep() вызываются tick() для устройств и спортивного оборудования,
  а также performStep() для жильцов и животных. Но WeatherService.tick() —
  метод, отвечающий за смену погоды каждые ~8 шагов (~2 часа) — НЕ вызывается.

ПОСЛЕДСТВИЕ:
  Погода устанавливается один раз в конструкторе WeatherService (случайным
  образом) и НИКОГДА не меняется за время всей симуляции. Механика смены
  погоды полностью не работает. Grandfather не реагирует на холодную погоду,
  так как COLD может выпасть только при запуске и не меняться.

ИСПРАВЛЕНИЕ:
  Добавить вызов ctx.getWeatherService().tick() в начало runStep().

СТАТУС: ИСПРАВЛЕНО
  + ctx.getWeatherService().tick(); добавлен в SimulationEngine.runStep()

БАГ #2 [КРИТИЧЕСКИЙ] — Fridge всегда в состоянии OffState (нет потребления)
────────────────────────────────────────────────────────────────────────────────
Файл:    src/.../devices/Fridge.java
Файл:    src/.../devices/Device.java

ОПИСАНИЕ:
  Абстрактный класс Device инициализирует состояние устройства через
  OffState() по умолчанию. Класс Fridge не меняет это состояние.
  Метод isInteractive() в Person помечает FRIDGE как неинтерактивное
  устройство, поэтому жильцы никогда не включают его случайно.
  Mother и Father обращаются к холодильнику через методы takeFood() /
  restockFull(), но turnOn() при этом не вызывается.

ПОСЛЕДСТВИЕ:
  accumulateConsumption() проверяет isOn() и возвращает 0 если устройство
  выключено. Холодильник — самый реальный постоянно-работающий прибор —
  НЕ показывает расход электричества ни в одном из отчётов.

ИСПРАВЛЕНИЕ:
  В конструкторе Fridge вызвать turnOn() сразу после инициализации запасов.

СТАТУС: ИСПРАВЛЕНО
  + добавлено turnOn() в конструктор Fridge после stock.putAll(INITIAL_STOCK)

БАГ #3 [СРЕДНИЙ] — Father.homeRoom скрывает (shadows) Person.homeRoom
────────────────────────────────────────────────────────────────────────────────
Файл:    src/.../people/Father.java
Файл:    src/.../people/Person.java

ОПИСАНИЕ:
  Класс Person объявляет поле: protected Room homeRoom;
  Класс Father объявлял приватное поле: private Room homeRoom;
  Это классическая проблема "field shadowing" в Java — два разных поля
  с одним именем в иерархии классов.

ПОСЛЕДСТВИЕ:
  Когда Father попадает в состояние "устал" и вызывает super.performStep(),
  Person.performStep() использует Person.homeRoom (начальная комната из
  конструктора), а не Father.homeRoom (последнее место перед уходом в магазин).
  Если Father двигался до похода в магазин, значения полей расходятся:
  - Person.homeRoom = LivingRoom (начальная, неизменная)
  - Father.homeRoom = текущая комната перед уходом

  После возвращения из магазина Father возвращается в Father.homeRoom,
  но при усталости Person отправляет его в Person.homeRoom. Логика
  разделяется между двумя разными полями.

ИСПРАВЛЕНИЕ:
  Удалить приватное поле homeRoom из Father, использовать унаследованное
  protected-поле из Person.

СТАТУС: ИСПРАВЛЕНО
  - Удалено поле "private Room homeRoom;" из Father.java
  - Удалено его присваивание в конструкторе Father
  - returnFromShopping() теперь явно использует локальную переменную
    для возвращаемой комнаты (чтение homeRoom → Person.homeRoom)

ЗАМЕЧАНИЕ #2 — EventBus не очищается при re-инициализации SmartHomeContext
────────────────────────────────────────────────────────────────────────────────
Файл:    src/.../house/SmartHomeContext.java

ОПИСАНИЕ:
  Метод initialize() очищает floors, residents, animals, eventLog, activityLog,
  consumptionLog. Но EventBus (private final) не очищается.
  Если initialize() вызывается повторно (например, при перезапуске

  слушателей (residents + SystemEventListener).

РЕКОМЕНДАЦИЯ:
  Добавить метод eventBus.clear() или пересоздавать EventBus при повторной
  инициализации. Также добавить метод clear() в EventBus.

СТАТУС: НЕ ИСПРАВЛЕНО (в текущей версии initialize() вызывается один раз)

ЗАМЕЧАНИЕ #6 — checkAndBuyDesires() имеет вероятность 31% вместо 30%
────────────────────────────────────────────────────────────────────────────────
Файл:    src/.../people/Person.java
Метод:   checkAndBuyDesires()

ОПИСАНИЕ:
  Условие: if (RANDOM.nextInt(100) > 30) return;
  RANDOM.nextInt(100) возвращает значения 0-99.
  "> 30" означает 31-99 = 69 значений → 69% пропуска → 31% выполнения.
  Если цель была "30% шанс", правильное условие: >= 70 для пропуска,
  или эквивалентно: if (RANDOM.nextInt(100) < 30) выполнять.

РЕКОМЕНДАЦИЯ:
  Если хотелось ровно 30%, использовать: if (RANDOM.nextInt(100) >= 30) return;

СТАТУС: НЕ ИСПРАВЛЕНО (разница 1%, влияние минимально)
